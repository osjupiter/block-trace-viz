Visualizing Linux Block I/O: How LVM, ZFS, and Btrfs Actually Write to Disk

Animated blktrace visualizations showing write amplification, COW behavior, and snapshot overhead across storage stacks.

What This Is

When you write 4KB to a file, how much actually hits the disk? The answer depends on your storage stack, and it can be surprising. We ran fio random writes through different Linux storage configurations inside a QEMU VM, captured every block I/O event with blktrace, and animated the results.

Each dot in the visualizations below is a real I/O completion event. The Y axis shows the physical sector on disk, and dots flow left-to-right over time. You can see exactly where and when each write (or read) lands on the physical device.

Experiment setup: QEMU/KVM VM with 4 vCPUs, 4GB RAM. Two 10GB virtio disks. I/O scheduler set to none, fio with --iodepth=1 --direct=1 for deterministic ordering. All I/O captured via blktrace on the physical block device.

Experiment 1: LVM Linear vs dm-thin

The simplest comparison: a plain LVM logical volume (linear mapping) vs a thin-provisioned LV. Same random 4KB writes, same seed, same duration.

LinearThin
Write events171,284 WS78,400 WS + 29,844 W
IOPS~17,000~10,500
Extra I/ONone~30k metadata writes (64KB chunks)

Linear is almost 1:1 — fio writes 4KB, the disk gets 4KB. Thin provisioning adds metadata writes (orange dots) for every new chunk allocation. The thin pool uses 64KB chunks, so each first-time write to a chunk triggers a metadata update.

The thin side also shows periodic ~100ms pauses roughly every second. These are dm-thin's transaction commits — it batches metadata btree updates and flushes them to disk, blocking data I/O during the commit.

Experiment 2: LVM Snapshot — COW vs ROW

What happens when you take a snapshot and keep writing? LVM's classic snapshots (dm-snapshot) use Copy-on-Write: before overwriting a block, it reads the old data and copies it to a COW area. Thin snapshots use Redirect-on-Write: new data simply goes to a new location.

The difference is dramatic. After the snapshot:

Linear + Snapshot (COW)Thin + Snapshot (ROW)
IOPS before snap~16,000~14,600
IOPS after snap~430~13,600
Slowdown37x1.07x
Read I/O after snapYes (COW read)None
LVM snapshot COW is catastrophically slow for random writes. Each 4KB write becomes a 6-step sequence: Read old block → Write to COW area → Flush → Write COW mapping table → Flush → Write new data. The two flushes (write barriers) ensure that the COW mapping is persisted before the original block is overwritten — without them, a crash could corrupt the snapshot. Measuring the actual timing of each step: R ~0.06ms, W ~0.08ms, Flush ~0.9ms × 2 = 1.8ms, total ~2ms per write. The flushes alone account for 90% of the latency, producing the 37x slowdown from ~16k IOPS down to ~430. The green dots (reads) are the COW source reads.
Why don't other snapshot implementations flush on every write? dm-thin and ZFS/Btrfs batch metadata updates into periodic transaction commits (dm-thin ~1s, ZFS ~5s, Btrfs ~30s). This means a crash could lose the last few seconds of writes, but snapshots remain consistent because the metadata is updated atomically at commit time. dm-snapshot chose the safer but slower path: every single COW operation is immediately persisted. An fsync() from the application would trigger a commit on the batched systems too, but fio with O_DIRECT alone does not force one.

Aside: What if we add fsync?

The 37x slowdown above is partly because dm-linear without snapshots never flushes the device cache — O_DIRECT alone doesn't trigger a flush. Adding fsync after every write (fio --fsync=1) forces all implementations to flush, leveling the playing field:

Without fsyncWith fsync
Linear~17,000 IOPS963 IOPS
Thin~10,500 IOPS410 IOPS
Linear + Snapshot~430 IOPS(already flushing)

With fsync, plain linear drops from 17k to 963 IOPS — close to dm-snapshot's 430 IOPS. The remaining 2x gap is the COW overhead (R + W + extra flush). dm-thin with fsync is even slower at 410 IOPS because each fsync triggers a full metadata transaction commit with 3 flushes. The takeaway: most of the "snapshot penalty" in Experiment 2 was actually the cost of flushing that other configurations were avoiding.

Is this a QEMU/virtio artifact? No. Running the same fio --fsync=1 directly on the host NVMe SSD produces 870 IOPS — nearly identical. The host SSD's fsync latency is ~1.1ms (median), and the write itself is only ~39μs. Flush accounts for 96% of the total latency. This is typical for consumer TLC/QLC NVMe SSDs; enterprise SSDs (e.g. Intel Optane) can achieve flush latencies of 10-50μs.

Thin snapshots use Redirect-On-Write (ROW) — instead of copying old data, new data goes to a fresh location and the metadata pointer is updated. However, look closely at the thin panel after the snapshot: there are green dots (reads) appearing. dm-thin's chunk size is 64KB, but fio writes 4KB. When a 4KB write hits a shared 64KB chunk, dm-thin must read the entire chunk, copy it to a new location, then apply the 4KB change — effectively COW at the chunk level. If the chunk size matched the write size (e.g. lvcreate -T ... -c 4k), these reads would disappear entirely.

Experiment 3: ZFS vs Btrfs — COW Filesystems

Both ZFS and Btrfs are copy-on-write filesystems. Every write goes to a new location, and metadata trees are updated with new pointers. But the implementation details lead to very different I/O patterns.

ZFSBtrfs
fio logical writes~450 MB~445 MB
Physical writes10,449 MB~445 MB
Write amplification~23x~1.0x
I/O patternSpread across entire 10GB diskSequential, single region

Why ZFS writes 23x more than requested

ZFS's default recordsize=128KB is the culprit. When fio writes 4KB to a record, ZFS COWs the entire 128KB record to a new location. That's a 32x data amplification before metadata is even counted.

Additionally, ZFS batches writes into Transaction Groups (TXGs) that flush approximately every 5 seconds. It ignores O_DIRECT and caches everything in the ARC, then writes large sequential batches across the disk's metaslab allocator. This is why the ZFS panel shows writes scattered across the full 10GB — the metaslab allocator intentionally distributes writes for wear leveling.

Setting recordsize=4k would eliminate the record-level amplification for this workload, but would hurt sequential read performance. This is a fundamental ZFS tradeoff for mixed workloads.

Why Btrfs is efficient here

Btrfs uses extent-based allocation. A 4KB COW write produces a 4KB extent at a new location — 1:1. With fallocate=none (no preallocation), every write is a fresh allocation, and the block allocator hands out sectors sequentially. The result is nearly zero write amplification and a perfectly sequential physical I/O pattern, even though fio is issuing random logical writes.

Note on fio settings: By default, fio preallocates the test file with fallocate() or sequential writes before starting random I/O. On Btrfs, this creates preallocated extents that make subsequent COW writes appear scattered (as they land near the original extent locations). We use --fallocate=none --create_on_open=1 to disable this and observe the true COW allocation pattern.

Experiment 4: Snapshots on COW Filesystems

Unlike LVM, ZFS and Btrfs are already COW. Does taking a snapshot change anything?

Three phases

  1. Phase 1 (10s): Fresh file, initial writes
  2. Phase 2 (30s): After snapshot #1 — old blocks pinned, cannot be reused
  3. Phase 3 (10s): After snapshot #2 — intermediate blocks from Phase 2 partially reclaimable

The I/O throughput barely changes across phases — no 37x slowdown like LVM. But the allocation pattern changes dramatically on Btrfs:

PhaseBtrfs sequential %Explanation
Phase 1100%Fresh file, no preallocation → allocator assigns blocks sequentially
Phase 2100%Snapshot pins all blocks → no reuse possible → pure sequential allocation
Phase 370%Phase 2's intermediate COW blocks (not pinned by snap1) become reusable
The hidden cost of COW snapshots: Performance doesn't drop, but disk consumption grows. Since old blocks can't be freed, every COW write consumes new space. ZFS at 96% pool usage + snapshots = writes stop entirely (ENOSPC). The "keep pools under 80%" rule exists for this reason.

Reproducing This

All experiments ran in a QEMU/KVM VM with two 10GB virtio disks. The full setup is in the repository.

The VM uses QEMU/KVM with Ubuntu 24.04, two 10GB virtio disks, 4 vCPUs, and 4GB RAM. Cloud-init provisions a dev user with build-essential, fio, blktrace, lvm2, zfsutils-linux, and btrfs-progs.

# Experiment scripts (run inside VM as root)
scripts/run_linear.sh       # LVM linear + snapshot
scripts/run_thin.sh         # LVM thin + snapshot
scripts/run_zfs.sh          # ZFS basic
scripts/run_btrfs.sh        # Btrfs basic (with --fallocate=none)
scripts/run_snap_long.sh    # ZFS/Btrfs with multiple snapshots

# All I/O captured with blktrace, parsed with blkparse,
# extracted to JSON with:
blkparse -i trace -q | awk '$6=="C" && $7~/[WR]/{
  printf "{\"t\":%.6f,\"s\":%d,\"n\":%d,\"type\":\"%s\"},\n", $4, $8, $10, $7
}'
